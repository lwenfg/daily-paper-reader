## 摘要
开发者如今从两个截然不同的来源获取代码：一是类似 Stack Overflow 的网站上累积的人类答案，二是大型语言模型的输出。我们围绕这一分化提出两个问题。第一，能否从代码片段本身恢复其来源；第二，对于同一任务，这两种来源采用的安全模式是否存在差异。仅使用开放源码，即一个开放权重语言模型的公共网关和公开的 Stack Overflow API，我们构建了一个完全可复现的流程，收集了 31 个安全敏感的编程任务的真实实现，包括带 PKCE 的 OAuth、JWT 验证、密码哈希和 SQL 访问等，这些实现来自 9 种语言模型和人类答案，并用确定性的安全与风格检测器对每个样本进行评分。在 528 个真实样本上，我们训练了一个交叉验证的分类器，区分人类与模型来源的准确率达到 93%，而基线为 78%；还训练了一个 7 路分类器，将样本归属于编写它的具体模型，准确率为 48%。接着，我们报告了来源在安全性上的分歧点，即哪些模式模型比人类语料库更常采用，以及哪些模式模型从人类语料库中继承。在 Python、JavaScript、Go 和 Java 中运行相同任务，我们发现安全性分歧在每种语言中都存在，而来源边界部分取决于语言，且在不同语言间并非对称迁移。一项漏洞修复案例研究中，我们向模型提供不安全的代码并要求其修复，在 21 个种子和 12 种弱点类别中，修复率达到 77%，但出现了一种反复出现的部分修复失败，即模型移除了不安全模式却未添加正确的防御措施。该流程由数据驱动，因此任何新任务或语言只需作为单一规范条目添加，并且一个故障关闭的检查器可根据存储数据重新推导本文中的每一个数字。

## Abstract
Developers now draw code from two very different sources, the accumulated human answers on sites such as Stack Overflow and the output of large language models. We ask two questions about that split. First, can the provenance of a code snippet be recovered from the code itself, and second, do the two sources differ in the security patterns they adopt for the same task. Using only open sources, a public gateway of open-weight language models and the public Stack Overflow API, we build a fully reproducible pipeline that collects real implementations of 31 security-sensitive programming tasks, among them OAuth with PKCE, JWT verification, password hashing, and SQL access, from 9 language models and from human answers, and scores every sample with deterministic security and style detectors. On 528 real samples we train a cross-validated classifier that recovers human versus model provenance with 93 percent accuracy against a 78 percent baseline, and a 7-way classifier that attributes a sample to the specific model that wrote it at 48 percent. We then report where the sources diverge on security, which patterns models adopt more often than the human corpus and which they inherit from it. Running the same tasks in Python, JavaScript, Go, and Java, we find the security divergence holds in every language while the provenance boundary is partly language-specific and does not transfer symmetrically between them. A vulnerability repair case study, in which models are handed insecure code and asked to fix it, finds a 77 percent repair rate across 21 seeds and 12 weakness classes, but a recurring partial-fix failure in which the model removes the insecure pattern without adding the correct defense. The pipeline is data driven, so any new task or language is added as a single specification entry, and a fail-closed checker re-derives every number in this paper from the stored data.

*   **来源与语言的关系**：
    *   **安全分歧是来源属性**：人类与模型之间的安全模式分歧在 Python、JavaScript 和 Go 中均存在，且模式相似，表明这是由于两种来源的根本性差异造成的，而非语言特例。
    *   **来源分类是语言部分相关的**：区分人机代码的决策边界部分依赖于编程语言。一个在 JavaScript 上训练的 P分类器在 Python 上表现尚可，但反过来则不行，这为部署此类分类器敲响了警钟。
*   **漏洞修复能力与缺陷**：
    *   模型在被明确要求修复漏洞时，表现出 **77%** 的修复成功率。
    *   **核心失败模式**：存在 **“部分修复”** 风险，即模型成功移除了不安全的代码模式（如 `md5`），但**未能添加正确的防御措施**（如 `bcrypt`），导致代码虽不再匹配已知漏洞特征，但实际仍不安全。这比完全不修复更具误导性。

### 7. 优点

*   **高度可复现的开放科学实践**：完全基于开放数据源和开放模型，并设计了故障关闭的数字校验器，极大提升了研究的可信度和可扩展性。
*   **创新的连接点**：巧妙地将**代码风格计量学**与**软件安全**两个领域结合，赋予了来源鉴别以实际的安全含义，即“鉴别来源是为了知道该重点检查什么漏洞”。
*   **现实且有深度的对比**：对比真实世界的人类代码库（而非人工编写的完美答案）和 LLM 输出，揭示了“两种来源失败模式不同”的深刻见解，指出其作为审查路由器的实用价值。
*   **可解释性强**：摒弃了复杂的深度学习模型进行来源鉴别，转而使用随机森林和可解释的轻量级特征，使得“为什么这样分类”一目了然（如代码长度和错误处理是主要区分点）。
*   **实验设计扎实全面**：数据驱动的管道、多维度的分析（任务级、模式级、模型级、语言级）以及正反案例的并置对比（如 Listings 对比），使得结论极具说服力。

### 8. 不足与局限

*   **安全检测的浅层性（构造效度威胁）**：其安全检测完全基于**正则表达式的词法匹配**。一个代码片段可能使用了 `bcrypt` 但配置错误，它仍会被标记为“安全”；或者代码虽然没有不安全模式，但逻辑上存在其他漏洞。论文坦诚指出，其度量的是“模式采纳”，而非“经过验证的安全性”。
*   **模型代表性有限**：所用模型全部为**开放权重模型**，未包含广泛使用的商业闭源助手（如 GitHub Copilot、ChatGPT 等）。因此，研究结论对当前市场主流的封闭模型不一定完全适用。
*   **数据集偏差**：
    *   **人类语料单一**：仅使用了 Stack Overflow，且其高票答案可能偏向于老旧实践，这本身就是其“安全得分低”的部分原因，但也是论文有意测量而非消除的现实点。
    *   **样本不均**：不同模型、不同语言的样本数量差异较大，可能导致个别模型或语言的评估结果噪声较大，但论文通过报告真实样本量来保持透明。
*   **场景的静态性**：
    *   **固定提示词**：实验使用了单一的、中性的提示词，未探索提示工程（如明确要求“编写安全代码”）对模型行为和安全得分的影响。
    *   **快照局限性**：所有结果都基于一个时间点，而 Stack Overflow 的内容会老化，模型也会不断更新，目前的发现会随时间推移而变化。
*   **归因的封闭世界假设**：七分类模型归因（归属到具体模型）是在已知的 9 个模型集合内进行的，无法应对开放世界中出现未知新模型的情况，其准确率会随模型池扩大和模型风格趋同而下降。
*   **对抗鲁棒性不足**：来源鉴别依赖于浅层风格特征，一个有意伪装代码格式的使用者能够轻易规避。
*   **算力细节缺失**：论文未提供推理阶段使用的具体硬件资源（GPU 型号、数量等），使得其他研究者难以精确评估复现该实验所需的计算成本。
