## 摘要
大型语言模型可以编写 SQL，但企业部署需要的不仅仅是看似合理的文本：输出必须语法正确，必须遵守每个角色和每个模式（schema）的策略，必须提供可证明的（而非尽力而为的）保证，不能随着生成长度增加而变慢，并且必须为每一个决策留下合规级别的记录。我们提出了 GRID（Grammar-Railed Decoding，语法约束解码），这是一种受语法约束的解码引擎，它将精确的下一个 token 掩码（mask）键控在解析器配置（词法分析器扫描状态 × LALR(1) 栈）上，而不是 token 序列上，并使用增量推进的 LALR(1) 解析器本身作为可行前缀预言器（viable-prefix oracle）。LLM token 通过字节级 trie 遍历桥接到语法终结符，其中通过上下文无关/上下文相关的划分，使得缓存键的健全性由构造保证。基于角色的访问控制被编译到语言中：角色投影限定了语法的产生式子集，模式词典限制了标识符终结符，因此在掩码级别上无法触及被禁止的动词和标识符。四个保证（健全性、完备性、终止性和近乎恒定的每 token 开销）均附有明确的前提条件，并且每个都配有测试或基准。Rust 内核使每个 token 掩码的中位数开销达到 3.6-6.7 微秒，在两个 token 化器上的 p50 和 p90 均优于 llguidance，且无误拒绝；在 n=16,000 时，每个 token 的守护开销是恒定不变的。在 Spider 数据集上，约束解码为 0.5B 模型带来了 +13 的执行准确度提升，并且对可证明掩码无法强制执行的残余部分（列级策略）进行一次检查器引导的修复传递，将 7B 模型提升到 94.5% 的可执行率。哈希链式每 token 审计追踪可逐位一致地重放，具备 100% 的篡改检测能力。我们明确指出了掩码无法做到的事情（分布忠实性、列级 RBAC、非 LALR(1) 语言），以及仍有开销的衡量之处。

## Abstract
Large language models can write SQL, but enterprise deployment demands more than plausible text: outputs must be syntactically valid, must respect per-role and per-schema policy, must carry provable (not best-effort) guarantees, must not slow down as generations grow, and must leave a compliance-grade record of every decision. We present GRID (Grammar-Railed Decoding), a grammar-constrained decoding engine that keys exact next-token masks on parser configurations (lexer scan state x LALR(1) stack) rather than on token sequences, and uses the incrementally advanced LALR(1) parser itself as a viable-prefix oracle. LLM tokens are bridged to grammar terminals by a byte-level trie walk with a context-independent/context-dependent split that makes cache-key soundness hold by construction. Role-based access control is compiled into the language: role projections subset the grammar's productions and schema lexicons restrict identifier terminals, so forbidden verbs and identifiers are unreachable at mask level. Four guarantees (soundness, completeness, termination, and near-constant per-token cost) are stated with explicit preconditions and each paired with a test or benchmark. Rust kernels bring the per-token mask to a 3.6-6.7 us median, ahead of llguidance at p50 and p90 on two tokenizers with zero false rejects; per-token guard cost is position-flat at n=16,000. On Spider, constrained decoding is worth +13 execution-accuracy points at 0.5B, and one checker-guided repair pass over the provably mask-unenforceable residue (column-level policy) lifts a 7B model to 94.5% executable. A hash-chained per-token audit trail replays bit-identically with 100% tamper detection. We state plainly what the mask cannot do (distribution faithfulness, column-level RBAC, non-LALR(1) languages) and where measured cost remains.

# GRID 论文总结：面向企业 SQL 生成的语法约束解码

## 1. 核心问题与整体含义
- **研究背景**：大语言模型（LLM）可以生成 SQL，但企业部署存在多重刚性需求：输出必须语法有效、遵守角色/模式策略、提供可证明的保证、不随生成变长而减速、并留下可审计的决策记录。现有基于 token 序列的约束方法（如语法掩码）无法同时满足这些需求。
- **整体含义**：GRID 提出一种以**解析器配置**（词法扫描状态 × LALR(1) 栈）为核心键控的解码引擎，将策略（RBAC）编译进语法，实现精确的、可证明的、恒定开销的约束解码，并为每一步生成留下哈希链审计记录，从而具备真正的企业级部署能力。

## 2. 方法论
### 核心思想
- **配置键控的掩码生成**：不再基于 token 序列缓存有效性，而是基于 **解析器配置（κ）** 决定下一 token 的可行集。配置即词法扫描状态（当前部分词素）和 LALR(1) 栈的组合，可唯一确定哪些 token 能延续成合法句子。
- **增量解析器即预言器**：利用 LALR(1) 解析器本身作为“可行前缀”的判定器，通过模拟规约链来精确给出允许的终结符集合，并判断允许 EOS（语句结束）的条件。
- **RBAC 编译进语法**：通过 L2（角色投影）剔除禁止的动词和子句，并通过 L3（模式词典）限制标识符（表名、列名）仅允许真实存在的合法词，这些被禁止的符号在掩码层面根本不可到达。
- **字节级 token-终结符桥接**：将 LLM 的 token 与语法终结符通过字节 trie 对齐，实现上下文无关（CI）与上下文相关（CD）的划分；CI 部分可缓存，CD 部分每次步基于实时栈裁决，确保缓存键的健全性。
- **四个可证明保证**（附测试验证）：  
  - **健全性**：每个发出的 token 都使输出保持在语言的可行前缀内。  
  - **完备性**：绝不屏蔽任何最终可生成合法句子的 token，永不进入“死胡同”。  
  - **终止性**：仅在完整句子处停止，且配备 Token 预留机制防止提前耗尽。  
  - **近乎恒定开销（需求 R）**：每 token 的守护开销是 O(1) 的，与生成长度无关。
- **审计链**：每一步记录配置哈希、掩码条目 ID、token 等，形成哈希链，可逐位重放并检测篡改。

### 关键技术细节
- **配置键与缓存策略**：缓存键由（类别、剩余字节、允许终结符签名、模式指纹）组成；跨角色共享时采用规范化扫描状态（genN），最大化共享并确保健全性（OBL-KEY1）。
- **CD 裁决等价类分组**：对跨边界 token 的裁决按终态集合分组，将每步数万条目压缩到约 150 个类别，大幅降低实时计算量。
- **Rust 内核演进**：从 v4 到 v7，逐步将掩码计算、CD 裁决、冷入口材料化等关键路径迁移至 Rust，消除 Python GC 干扰，冷窗争用大幅改善。
- **服务契约**：掩码计算与 GPU 前向传播重叠；采用“跳过本轮”延迟机制，当掩码未就绪时仅延迟该请求，不阻塞同批其他请求，也绝不使用近似掩码。

## 6. 主要结论与发现
- **性能领先**：在每 token 掩码延迟上，GRID 在两个常用 token 化器上均优于 llguidance 和 XGrammar（p50: 3.6µs vs 6.6µs / 14.9µs），且零误拒，同时每 token 开销是恒定不随位置增长的（需求 R 成立）。
- **语法与策略强制**：RBAC 编译进语法使得禁止的动词、标识符在掩码级别绝不出现，无任何绕过；列级策略因上下文敏感性证明无法由掩码强制，但由后置检查器捕获并用于修复循环。
- **显著提升下游任务**：在 Spider 上，约束解码可为 0.5B 小模型带来 +13 执行准确点；检查器引导的修复将 7B 模型可执行率推至 94.5%，相比无约束提升 2.3 个 EX 点。
- **审计可追溯**：哈希链审计记录可在依赖件存档后逐位一致重放，100% 检测篡改。
- **诚实边界**：论文明确列出了掩码无法保证的方面（分布忠实性、列级 RBAC、非 LALR(1) 语言）、残余开销（冷 trie 遍历、冷窗争用）以及外部环境引入的 artifacts，不虚构能力。

## 7. 优点
- **理论严密**：基于解析配置的掩码使保证成为定理，且每个保证配有针对性的测试；对缓存键健全性给出了形式化的义务并加以验证。
- **工业化设计**：将 RBAC、模式迁移、冷启动、批处理服务、审计全部纳入统一的架构，Rust 内核从 v4 到 v7 逐步挖潜，真实环境服务开销低至 +1.51%。
- **问题定义清晰**：明确了掩码能做什么、不能做什么，并对不可掩盖的残余（列级策略）提供了后置修复的闭环。
- **评测扎实**：引擎、规模、ML 任务、服务、策略、审计多维度覆盖，且对每一个代价来源（冷窗争用、GC 影响）进行了逐层分解，超越简单的排行榜比较。

## 8. 不足与局限
- **语言范围**：仅支持 LALR(1) 可解析的语言，对 JSON Schema 中的 allOf/if-then-else 等构造会有编译错误（MaskBench 中 79/315 未能编译），未实现 Earley 回退。
- **分布忠实性**：掩码重归一化会改变采样分布，未提供分布级校正，对小模型质量损失已有记录。
- **冷窗开销**：首次遇到全新 schema 时的冷 trie 遍历和编译会带来约 0.66s 的窗口，此刻会因 CPU/内存带宽争用导致同批其他请求有约 34% 的短暂额外延迟，需依赖并行度与调度 nice 缓解，尚未彻底消除。
- **列级 RBAC**：只能后置处理，掩码阶段无法强制，依赖下游检查器，若后置环节缺失则成为漏洞。
- **服务特性依赖**：部分极端延迟来自上游引擎（vLLM 多进程拓扑的冻结步），虽已归档为外部 artifact，但仍影响服务 p99。
- **冷编译/缓存瞬态**：冷启动编译时间（约 27.3ms）和 MaskBench 一次性的冷遍历尾巴，对于一次性的短请求不友好。
