## 摘要
具有丰富静态语义的语言（如 Rust）为 AI 生成的代码提供了更强的保证，但其严格性增加了生成难度。现成的编译器可以在生成后提供有价值的反馈，但无法指导中间生成步骤，例如自回归 LLM 解码过程中的步骤。约束解码通过采样时拒绝无效标记而更早介入，但需要白盒模型访问权限，并且为语义约束重新实现成本高昂。我们引入了生成式编译，这是第一种在生成过程中对部分程序获取编译器反馈的方法。核心技术装置是 sealor：一种轻量级、主要由语法引导的转换，将部分程序转换为标准编译器可以诊断的完整程序。其设计使得可能完成的部分程序永远不会被拒绝，同时保留足够的代码上下文以尽早捕获真正的死胡同。我们在一个类似 Rust 的核心演算上构建了这样的 sealor，并证明它满足这些性质，全部在 Lean 中进行了机械化验证。我们将其扩展为首个针对真实 Rust 的部分程序检查器。我们在具有挑战性的仓库级 Rust 编码任务上评估了我们的方法，涵盖前沿的黑盒模型和开放权重模型。结果表明，与标准的生成后反馈相比，生成式编译减少了无法编译的输出，并提高了功能正确性。它通过在错误源头附近、生成早期检测到广泛的错误，从而减少了错误级联并实现了有针对性的诊断。更广泛地说，生成式编译是朝着使编译器成为 AI 辅助编程中在生成期间就活跃的一等公民，而非独立的生成后检查，迈出的一步。

## Abstract
Languages with rich static semantics, such as Rust, provide stronger guarantees for AI-generated code, but their strictness makes generation more difficult. Off-the-shelf compilers can provide useful feedback post-generation, but does not guide intermediate generation steps, such as those during autoregressive LLM decoding. Constrained decoding intervenes earlier by rejecting invalid tokens during sampling, but requires white-box model access and costly reimplementation for semantic constraints.We introduce generative compilation, the first approach to obtaining compiler feedback on partial programs during generation. The core technical device is a sealor: a lightweight, mostly syntax-guided transformation that converts partial programs into complete ones that standard compilers can diagnose. It is designed such that possible-to-complete partial programs are never rejected, while preserving enough code context to catch genuine dead ends early. We construct such a sealor on a core Rust-like calculus and prove that it satisfies these properties, all mechanized in Lean. We extend it to the first partial-program checker for real Rust. We evaluate our method on challenging repository-level Rust coding tasks, across both frontier black-box and open-weight models. We show that generative compilation reduces non-compiling outputs and improves functional correctness, relative to standard post-generation feedback. It does so by detecting a broad range of errors close to their source and early during generation, thereby reducing errors cascades and enabling focused diagnostics. More broadly, generative compilation is a step toward making compilers a first-class citizen of AI-assisted programming active during generation, rather than a separate post-generation check.

### 方法论

论文提出的核心方法是“生成式编译”，其关键技术装置是一个名为 **sealor（密封符）** 的转换工具。

-   **核心思想**：
    -   在LLM逐个token生成代码的过程中，将当前的“部分程序”作为输入，通过sealor自动补全缺失的语法结构并插入类型正确的占位符，将其“密封”成一个语法完整、可以被标准编译器（如`rustc`）检查的完整程序。
    -   通过编译这个密封后的程序，获取诊断信息，再将其映射回原始的部分程序，从而实现对尚未生成完毕的代码进行即时错误检查。

-   **关键技术细节**：
    -   **sealor的性质**：为了实现可信的即时反馈，sealor的设计需满足两个核心性质（在FR演算中得到了形式化证明）：
        -   **完整性**：如果原始的部分程序存在某种有效的补全方式，那么密封后的程序也必须是有效的。这保证了sealor不会“误报”错误。
        -   **选定的可靠性**：如果密封后的程序是有效的，那么原始的部分程序至少在某些特定类别下（如语句边界）存在有效的补全方式。这保证了sealor的接受是有根据的。
    -   **sealor的设计原则**：论文采用了“语法引导”的轻量级设计，主要通过语法结构决定如何密封，而不是重新实现一个完整的类型检查器。
        1.  **保留完整结构**：对于已生成完毕的代码部分，sealor会原样保留。
        2.  **抽象化未完成部分**：对于未完成的部分，sealor会用精心设计的占位符替换，例如：
            -   `𝜀` / `hole val()`：代表一个类型正确的任意值。
            -   `hole div()`：代表一个分歧（如`panic!()`）的语句，可以满足任意返回类型，并让编译器跳过之后的借用检查，防止因部分代码暂时不可达而产生的误报。
        3.  **根据上下文处理**：例如，对于未完成的`if`或`while`循环体，sealor会补上`hole div()`占位符，使其成为分歧块，从而能检查已生成的条件和部分代码的正确性。
    -   **算法流程**：
        1.  LLM通过流式（streaming）API逐段生成代码。
        2.  生成编译模块（如图6右侧所示）采用“最新优先”策略，获取最新的代码前缀。
        3.  调用sealor将此前缀密封为完整程序。
        4.  调用标准编译器C检查密封后的程序，得到判决和诊断信息。
        5.  若编译失败，将错误诊断从密封程序映射回原始部分程序，并与提示词一起反馈给LLM，让LLM重新从错误源头开始生成或修改。

### 不足与局限

-   **理论层面**：
    -   对sealor性质的证明是在轻量化的Featherweight Rust（FR）演算上完成的，对于真实Rust的扩展部分采用了非形式化的论证，其正确性依赖于sealor实现与`rustc`行为的对齐，而非严格的形式化验证。
    -   方法对“诊断信息`err`”本身的质量和准确性（如是否真的是原始代码的缺陷而非密封引入的）没有形式化建模，仅在实践中观察到良好表现。
-   **方法层面**：
    -   sealor的规则是手工编写的，这限制了该方法向其他语言和新语言特性快速扩展的能力。
    -   为保证完整性，sealor有时会过于保守地放过一些错误（例如对未完成的字段访问的处理），需要等到代码完成后再捕获。
-   **应用层面**：
    -   文中将生成式编译与一个固定的重启机制耦合，探索更灵活的调用策略（如交由编码Agent决策、用于训练等）是未来的工作。
    -   该方法的效果仍然依赖于模型理解和使用编译反馈的能力，对于能力较弱的模型，其优势可能有限。
